# Advanced State Management в React

Ключевая особенность состоит в том, что в SPA-приложении разным веткам _Component Tree_ может быть нужен доступ к общему состоянию. Предположим, что мы разрабатываем online-магазин и в это магазине у нас есть корневой элемент App, у которого есть две дочерние ветки:

- App -> Shop -> Product
- App -> Header -> CartModal -> Cart

Поскольку изменение корзины товаров осуществляется в Product, а отображение корзины в Cart (т.е. в разных ветках), мы вынуждены хранить состояние на самом верхнем уровне - в App. Это значит, что в обычном подходе нам потребуется транслировать состояние "Cart State" с уровня App через все компоненты до Product и Cart, что приведёт к тому, что все компоненты будут участвовать в передаче состояния до самого нижнего уровня, т.е. все компоненты будут затронуты.

Передача разделяемых данных через множество уровней компонентов (_components layers_) называется **Prop Drilling**. При этом, абсолютному большинству компонентов, эти данные вообще не нужны и не интересны - они просто передают эти данные своим дочерним компонентам.

>И вот именно Prop Drilling является основной проблемой, т.к. это приводит к тому, что компоненты становятся более сцепленными друг с другом, что критически нарушает гибкость изменения приложения и снижает возможность повторного использования компонентов.

## Концепция Component Composition

Идея состоит в том, что если в типовом подходе у нас используется цепочка вложенных компонентов, то для устранения зависимостей мы выносим элемент нижнего уровня, который работает с состоянием максимально близко к App-у, т.е. к хранилищу состояния.

В коде это начинает выглядеть следующим образом - в компоненте "App.js" мы напрямую можем передавать состояние Product-у:

```js
<Shop onAddItemToCard={handleAddItemToCart}>
    {DUMMY_PRODUCTS.map((product) => (
        <li key={product.id}>
            <Product {...product} onAddToCart={onAddItemToCart} />
        </li>
    ))}
</Shop>
```

Т.е. Shop-компонент всё ещё wrapper для Product, но он уже исключен из трансляции состояния.

Однако, этот подход имеет очень ограниченные возможности применения, из-за чего его следует рассматривать как "костыль", применимый лишь в некоторых случаях.

## React's Context API

Задача этого API состоит в возможности совместного использования данных между компонентами, но без необходимости "пробрасывания свойств". По сути, API реализует шаблон проектирования publish/subscribe, или, как его ещё называют, Observer.

Идея шаблона состоит в том, что создаётся некоторый wrapper (в общем случае, выше компонента App, но это не обязательно) к которому привязывается состояние (например, Cart State). Т.е. хранилищем состояния перестаёт быть App, а передача этого разделяемого состояния перестаёт осуществляться через Props. Те компоненты, которым нужен доступ к разделяемому состоянию получают этот доступ напрямую, без привлечения родительских компонентов в качестве посредников.

Общая рекомендация - создавать артефакты, имеющие отношение к Context-у в папке "store", или "contexts". В этой папке добавляется файл с суффиксом "-context", например: "shopping-cart-context.jsx". В таком файле создаётся контекст (структура с данными, которые будет разделяться между разными компонентами), а сам контекст может быть описан либо здесь же, либо в отдельной папке "types":

```js
import { creareContext } from 'react';

export const CartContext = createContext({
    items: []
});
```

После этого нужно обернуть компоненты, для которых контекс имеет смысл, в созданный нами контекст. Иногда это делается в "App.js". Сначала импортируется определённый контекст:

```js
import { CardContext } from './store/shopping-cart-context.jsx';
```

Затем компоненты верхнего уровня включаются в этот, представленный уже в виде JSX-компонента, контекст:

```js
<CardContext>
    <Header ... />
    <Shop>
        ...
    </Shop>
</CardContext>
```

Следует заметить, что приведённый выше синтаксис прекрасно работает в React 19, но в более старых версиях нужно использовать свойство "Provider", объекта, который создаётся посредством вызова createContext(). В нашем случае компонент будет называться "CardContext.Provider". Заметим, что в свойствах wrapper-а мы определяем поле value, которое будет осуществлять инициализацию контекста:

```js
<CartContext.Provider value={{items: []}}>
```

>Установка параметра value - это полноценная инициализация, а определение полей в `CartContext = createContext()` это подсказка для autoCompletion.
>
>В корректно написанном коде нужно передавать не пустые данные, а состояние:
>
>```js
>function App() {
>   const [shoppingCart, setShoppingCart] = useState({ items: [] });
>```
>
>```js
><CartContext.Provider value={shoppingCart}>
>```

### Использование контекста (Consuming the Context)

Чтобы использовать контекст необходимо импортировать его в дочернем компоненте-потребителе:

```js
import { CardContext } from './store/shopping-cart-context.jsx';
```

А также загрузить один из двух hook-ов: **useConext**, или **use** (он действительно называется просто use). Хук **use** более гибкий, его можно, в частности, использовать в условных операторах. Однако use() может быть применим только в React 19.

Пример:

```js
import { useContext } from 'react';
```

Далее необходимо сгенерировать объект доступа к контексту, используя hook и имя контекста:

```js
export default function Cart({item, onUpdateItemQuantity}) {
    const cartCtx = useContext(CartContext);
}
```

Следует заметить, что при вызове useContext() мы можем использовать destructuring:

```js
const { items } = useContext(CartContext);
```

После этого мы можем обращаться к полям контекста по имени cartCtx, например:

```js
<div id="cart">
    {cartCtx.items.length === 0 && <p>No items in cart!</p>}
```

## Обновление контекста

Обновление контекста более сложная операция, чем его использование. Общий подход состоит в том, что создаётся специальный объект, в котором есть данные и метод их изменения, например:

```js
const ctxValue = {
    items: shoppingCart.items,
    addItemToCart: handleAddItemToCart // Ссылка на функцию, которая обновляет контекст
}
```

Соответственно, именно этот объект используется в качестве контекста:

```js
return (
    <CardContext.Provider value={ctxValue}>
        <Header>
            ...
        </Header>
        <Shop>
            ...
        </Shop>
    </CardContext.Provider>
)
```

Соответственно, в дочерних компонентах мы можем определить `useContext(CardContext)` и через него получить доступ к функции addItemToCart(). Другими словами, мы включаем в контекст ссылку на функцию root-компонента, которая работает с состоянием. Контекст достаавляет эту ссылку в любой дочерний компонент и уже оттуда можно эффективно и безопасно изменять этот состояние.

## Альтернативный подход

Кроме описанного выше подхода можно использовать ещё и свойство `Consumer` компонента-обёртки. В нашем случае, это `CardContext.Consumer`:

```js
<CardContext.Consumer>
    {(cardCtx) => {
        // Дочерние компоненты - верстка
    }}
</CardContext.Consumer>
```

Однако этот подход сложнее, чем типовой подход.

## Извлечение кода для управления Store из компонента

Код управления состоянием (Store/Context) может быть достаточно объёмным - сотни строк. По этой причине, обычно разработчики выделяют этот код из компонента и создают отдельный класс/компонент, xxxContextProvider, в котором и реализуют основные функции для управления контекстом. Реализация часто создаётся по следующей схеме: файл с компонентом называется "xxx-context.jsx", в нём содержатся высокоуровневые функции для управления контекстом, а рендеринг выглядит как:

```js
return 
    <CardContext.Provider>
        {children}
    </CardContext.Provider>;
```

Таким образом осуществляется отделение кода управления состоянием от верстки компонентов, использующих этот контекст.

Реализация "ThemeContext.tsx" может выглядеть следующим образом:

```tsx
import React, { createContext, useContext } from "react";

type Theme = "light" | "dark";  // Определяем тип Theme

interface ThemeContextValue {   // Определяем интерфейс контекста
  theme: Theme;
  toggleTheme: () => void;
}

// Создаём контекста
const ThemeContext = createContext<ThemeContextValue | null>(null);

// При определении провайдера, описываем интерфейс Props, поскольку
// через него передаются дочерние компоненты, а также тема интерфейса
// "по умолчанию"
interface ThemeContextProviderProps {
  children: React.ReactNode;
  defaultTheme?: Theme;
}

export class ThemeContextProvider 
  extends React.Component<ThemeContextProviderProps, { theme: Theme }>
{
  constructor(props: ThemeContextProviderProps) {
    super(props);
    this.state = {
      theme: props.defaultTheme ?? "light",
    };
  }

  toggleTheme = () => {
    this.setState((prev) => ({
      theme: prev.theme === "light" ? "dark" : "light",
    }));
  };

  render() {
    // Значение оборачивается в объект вручную, без useMemo (класс-компонент)
    const contextValue: ThemeContextValue = {
      theme: this.state.theme,
      toggleTheme: this.toggleTheme,
    };

    return (
      <ThemeContext.Provider value={contextValue}>
        {this.props.children}
      </ThemeContext.Provider>
    );
  }
}

// Опционально: hook для удобного доступа к контексту
export function useTheme(): ThemeContextValue {
  const context = useContext(ThemeContext);
  if (!context) {
    throw new Error("useTheme must be used within a ThemeContextProvider");
  }
  return context;
}
```

В реализации импорт useContext нужен только для hook-а useTheme, который является важным дополнением в файле "ThemeContext.tsx". Он простой и удобный в использовании в классах потребителях:

```ts
export function useTheme(): ThemeContextValue {
  const context = useContext(ThemeContext);
  if (!context) {
    throw new Error("useTheme must be used within a ThemeContextProvider");
  }
  return context;
}
```

Опционально мы моглиы бы экспортировать ещё и контекст, если бы где-то в коде был нужен прямой доступ к нему:

```tsx
export { ThemeContext };
```

По коду решения в целом, следует заменить, что у нас есть класс ThemeContextProvider, который реализует провайдера контекста в приложении. Этот компонент используется в "App.tsx" для формирования области видимости контекста. Одновременно с этим, в реализации ThemeContextProvider у нас встречается строка ThemeContext.Provider, которая использует компонент Provider, являющийся частью объекта ThemeContext, полученного на фабрике контекстов `createContext<>`. Различие в написании - буквально один дополнительный символ, но это два абсолютно разных компонента/класса.

## Что такое useReducer()

**useReducer** — встроенный хук React для управления состоянием в функциональных компонентах. Он служит альтернативой useState, особенно когда логика работы с состоянием становится сложной. Когда использовать useReducer вместо useState?

- Состояние сложное. Оно состоит из множества подсостояний или вложенных объектов
- Логика обновления состояния нетривиальна. Новое состояние зависит от предыдущего, и логика не сводится к простому присваиванию
- Есть несколько способов изменить состояние. Множество действий (actions) делают код с useState запутанным. useReducer позволяет централизовать логику в одном редукторе
- Вы хотите предсказуемости. Подход «действие → редуктор → новое состояние» делает переходы состояний более явными и предсказуемыми
- Планируете масштабирование. Код с useReducer легче расширять и поддерживать при росте сложности состояния

Типовой подход в промышленных приложениях состоит в использовании нескольких функций-редукторов. Можно создать несколько хуков, либо осуществлять комбинирование редукторов, как это сделано, например, в Redux. Вполне можно использовать Context API с несколькими редукторами.

Экземпляр хука можно получить так:

```js
const [state, dispatch] = useReducer(reducer, initialArg, init);
```

Он возвращает массив из двух элементов:

- state — текущее состояние
- dispatch — функция для отправки действий (actions), которые изменят состояние

**dispatch** — это функция, которую вы вызываете, чтобы сообщить редуктору о необходимости обновить состояние:

```js
dispatch({ type: 'SOME_ACTION', payload: /* данные */ });
```

Вызов dispatch приводит к:

- Передаче объекта действия в редуктор
- Вычислению нового состояния редуктором
- Обновлению состояния компонента и его перерисовке

Параметры useReducer:

- reducer (обязательный) — функция-редуктор. Принимает текущее состояние (state) и действие (action), возвращает новое состояние
- initialArg (обязательный) — начальное значение состояния
- init (опциональный) — функция инициализатора. Используется для ленивой инициализации состояния: если передана, начальное состояние вычисляется как init(initialArg)

**Редуктор** — чистая функция со следующей сигнатурой:

```js
function reducer(state, action) {
  // логика обновления состояния
  return newState;
}
```

- state — текущее состояние
- action — объект действия. Обычно содержит свойство type (тип действия) и опциональные данные (payload)

Функция должна вернуть новое состояние, а не мутировать старое.
